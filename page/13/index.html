状态模式（行为模式五）

2016-05-08

状态模式：状态模式又称为类的状态模式，描述的是一个类根据不同的状态处理不同的业务逻辑，通常我们在项目中会通过 if(state) else,或switch来处理，但是对于复杂的类来说这样的代码显然可读性差，而且耦合性高。使用状态模式可以把每个类对应的不同状态交给具体的类处理接口来处理。

类图

分析

状态模式组成：

  1. 抽象状态(StateHandle） 处理接口。
  2. 具体状态(ConcreteState)处理接口。
  3. 具体类(Content)。

具体类处理者对细节的处理交给抽象状态处理者的实现，替代if-else 逻辑耦合，客户端只需要调用具体类的 hand方法就可以，屏蔽状态处理的细节。

代码

顶层抽象类

抽象类的实现

public class ConcreteState implements StateHandle {
    @Override
    public void doAction(String state) {
        System.out.println("this request to concrete StateHandle "+state);
    }
}

状态类 

public class Content {
    private StateHandle state;
    private String stateFlag;

    public String getStateFlag() {
        return stateFlag;
    }

    public void setStateFlag(String stateFlag) {
        this.stateFlag = stateFlag;
    }

    public StateHandle getState() {
        return state;
    }

    public void setState(StateHandle state) {
        this.state = state;
    }
    public void HandleState(String stateFlag){
        state.doAction(stateFlag);
    }

}

APP类调用 

public class App {
    public static void main(String[] args) {
        StateHandle stateHandle=new ConcreteState();
        Content content=new Content();
        content.setState(stateHandle);//注入具体的处理类
        content.setStateFlag("Ready");
        content.HandleState(content.getStateFlag());//调用具体处理类

    }
}

优点

状态模式相对于 if -else只是把 if-else 对应的代码交给具体的状态处理着，避免了写一堆逻辑代码，代码结构清晰。解耦了逻辑处理与业务判断的耦合性，方便扩展。

中介者模式（行为模式四）

2016-05-07

中介者模式把类之间复杂的通信关系或复杂的关系引用，交由一个第三方的中介者类来处理，类之间的通信通过这个中介来维护，便于系统扩展。

类的结构图

分析

中介模式的组成：

  1. 抽象的中介（Mediator）,定义中介处理消息的具体操作。
  2. 抽象的中介使用者类，也叫同事类（Person）利用中介来发送消息。
  3. 具体中介（ConcreteMediator）,具体的消息处理者 。
  4. 具体的的中介使用者（PersonA,PersonB），利用中介来处理具体业务。

优点

把系统间的通信交给抽象的第三方中介，解耦了系统，便于系统的维护和扩展。

缺点

在具体中介者类中包含了大量中介使用者之间的交互细节，可能会导致具体中介者类非常复杂，使得系统难以维护。

迭代器模式（行为模式三）

2016-05-05

迭代器模式面对的是容器，为容器提供一系列的遍历，删除操作，作为客户并不直接操作容器，迭代器类为容器提供统一接口,遍历容器中的数据。

类图

分析

迭代器的模式的组成：

  1. 抽象遍历接口，顶层Iterator提供统一容器遍历接口。
  2. 抽象容器（容器中不包含任何数据结构），定义容器需要的迭代器。
  3. 具体的迭代器 遍历具体的容器（内部类）。
  4. 具体容器定义具体的数据结构，对客户端提供一个遍历接口。

优点

迭代器模式把对象的遍历，和对象存放的容器解耦了，对容器的扩展很方便，对外提供了统一的遍接口，在JDK中应用广泛如Iterator接口，适用于遍历容器。

缺点

系统每增加一个容器，就需要增加一个迭代器类，增加了系统的复杂性。

观察者模式（行为模型二）

2016-05-04

观察者模式由：抽象被观察Object,抽象观察者Observer,具体被观察者ConcereteObject,具体观察者ConcreteObserver组成。当具体的Object发生变化时，通知所有观察者。
参考

类结构图

分析

顶层定义抽象的观察着，Observer 当具体对象发生变化的时候调用所有具体的观察者。抽象的被观察者Subject 实现了一些通用的操作观察者的方法，具体的被观察者发生变化时候调用观察者。

优点

观察者模式可以实现表示层和数据逻辑层的分离，并定义了稳定的消息更新传递机制，抽象了更新接口，使得可以有各种各样不同的表示层作为具体观察者角色。

缺点

如果一个观察目标对象有很多直接和间接的观察者的话，将所有的观察者都通知到会花费很多时间。

如果在观察者和观察目标之间有循环依赖的话，观察目标会触发它们之间进行循环调用，可能导致系统崩溃。

观察者模式没有相应的机制让观察者知道所观察的目标对象是怎么发生变化的，而仅仅只是知道观察目标发生了变化。

命令模式（行为模式一）

2016-05-03

命令模式的本质是把一条条命令封装成一个对象，客户端不需要知道具体的命令是如何实现的，实现调用者和实现者解耦，命令模式中的所谓宏命令就是一系列命令的一个组合，实现起来方便。

类的结构图

分析

Car 这个对象中有三个具体的方法（run,sppedup,stop），使用Command抽象成三条具体的命。使用Invoker来封装三条具体的命令，客户端只需要调用Invoker类，达到命令调用者与具体的实现类相解耦的目的。

优点

降低系统的耦合度。
新的命令可以很容易地加入到系统中。
可以比较容易地设计一个命令队列和宏命令（组合命令）。

缺点

使用命令模式可能会导致某些系统有过多的具体命令类。因为针对每一个命令都需要设计一个具体命令类，因此某些系统可能需要大量具体命令类，这将影响命令模式的使用。
